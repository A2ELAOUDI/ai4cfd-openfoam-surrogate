## 1. Physical Problem

### 1.1 Dam-Break Scenario

The dam-break problem is a canonical benchmark in computational fluid dynamics.
At time t = 0, an instantaneous gate release liberates a water column of height
H and width W confined against a left wall. Under gravity, the water propagates
as a bore across an initially dry domain until it impacts the right wall and
reflects.

**Geometric configuration:**

```
┌────────────────────────────────────────┐ y = 2.0 m
│                 air                    │
│                                        │ (atmosphere, open)
│ █████                                  │
│ █████  H                               │
│ █████◄─── water column                 │
│ ██W██                                  │
└────────────────────────────────────────┘ y = 0 m (no-slip floor)
x=0 (left wall, no-slip)             x=4.0 m (right wall, no-slip)
```

The parameter space explored in this project:

| Parameter     | Symbol | Range         | Unit |
|---------------|--------|---------------|------|
| Water height  | H      | 0.40 – 0.80   | m    |
| Water width   | W      | 0.30 – 0.60   | m    |
| End time      | T      | 0 – 2.5       | s    |

### 1.2 Why Dam-Break?

- **Well-studied:** Ritter (1892) provides an analytical shallow-water solution
  for validation.
- **Computationally tractable:** A single 2D case completes in minutes on a
  desktop CPU with interFoam.
- **Rich dynamics:** The problem involves free-surface evolution, impact events,
  and wave reflection — representative of many engineering flows.
- **Parameterisable:** Two geometric parameters (H, W) cleanly define the
  initial condition, making dataset generation straightforward.

## 2. Governing Equations

### 2.1 Volume of Fluid (VOF) Method

The water–air interface is tracked implicitly via the phase indicator field
α (alpha.water), which satisfies:

```
∂α/∂t + ∇·(αU) + ∇·[α(1-α)Uᵣ] = 0
```

where:
- α ∈ [0, 1]: water volume fraction (0 = air, 1 = water)
- U: mixture velocity [m/s]
- Uᵣ: interface compression velocity (anti-diffusion term, MULES approach)

The mixture density and viscosity are computed as:

```
ρ = αρ_water + (1−α)ρ_air
μ = αμ_water + (1−α)μ_air
```

### 2.2 Incompressible Navier-Stokes

The mixture momentum equation is solved for incompressible flow:

```
∇·U = 0                                         (continuity)

∂(ρU)/∂t + ∇·(ρUU) = −∇p* − g·x∇ρ + ∇·(μ_eff ∇U)   (momentum)
```

where p* = p − ρg·x is the modified pressure (excess over hydrostatic),
and μ_eff includes the laminar viscosity only (laminar model assumed).

### 2.3 Analytical Reference (Ritter Solution)

For a frictionless flat bed, Ritter (1892) gives the wave front position as:

```
x_front(t) = W + 2√(gH) · t
```

and the velocity at the wave front:

```
U_front = 2√(gH)
```

This is used as a baseline to validate the numerical results and to inform
the physical plausibility of the surrogate predictions.

## 3. Dataset Generation Strategy

### 3.1 Parameter Sampling

Cases are generated by Latin Hypercube Sampling (LHS) over the 2D parameter
space (H, W). LHS ensures uniform coverage of the space with fewer samples
than a full grid — important when each case takes O(minutes) to simulate.

For the included sample_data, three cases were manually chosen to span
the parameter range:

| Case | H [m] | W [m] | √(gH) [m/s] |
|------|-------|-------|-------------|
| 001  | 0.50  | 0.40  | 2.21        |
| 002  | 0.65  | 0.45  | 2.53        |
| 003  | 0.75  | 0.55  | 2.71        |

### 3.2 OpenFOAM Workflow

For each (H, W) pair:
1. Copy base_case directory → case_XXX/
2. Patch `blockMeshDict` (domain geometry)
3. Patch `setFieldsDict` (water column coordinates)
4. Run: `blockMesh` → `setFields` → `interFoam`

Output: time directories at every 0.1 s, containing `alpha.water` and `U`.

### 3.3 Field Extraction

From each time directory:
- **wave_front_x**: maximum x-coordinate of cells where α > 0.5
- **max_velocity**: maximum |U| over the entire domain

These two scalars become the surrogate outputs.

### 3.4 Dataset Assembly

Each case contributes 26 rows (t = 0.0, 0.1, …, 2.5). Each row has:
- **Features x = [H, W, t]** (3 inputs)
- **Targets y = [wave_front_x, max_velocity]** (2 outputs)

Features and targets are normalised independently to [0, 1] via min-max
scaling (statistics computed over the full dataset).

Split: 70% train / 15% validation / 15% test (random, stratified by case).

## 4. Surrogate Model Architecture

### 4.1 Design Choices

A multilayer perceptron (MLP) was chosen because:
- **Input dimensionality is low** (3 features): no spatial structure to
  exploit; convolution/GNN adds no benefit.
- **Dataset is small** (O(100) points): complex architectures overfit.
- **Interpretability**: weight magnitudes reveal relative importance of
  H, W, and t.
- **Speed**: inference is O(μs), suitable for real-time design exploration.

### 4.2 Architecture

```
Input [3]
  │
  ├─ Linear(3 → 64)  + BatchNorm(64) + ReLU + Dropout(0.1)
  │
  ├─ Linear(64 → 128) + BatchNorm(128) + ReLU + Dropout(0.1)
  │
  ├─ Linear(128 → 64) + BatchNorm(64) + ReLU + Dropout(0.1)
  │
  └─ Linear(64 → 2)
       │
  Output [2]:  [wave_front_x_norm, max_velocity_norm]
```

Total parameters: ~14,000 (lightweight by design).

BatchNorm stabilises gradients for the small batch sizes forced by the
limited dataset. Dropout(0.1) provides mild regularisation.

### 4.3 Physics-Informed Loss

The training objective is:

```
L = MSE(ŷ, y) + λ · L_physics
```

where:
```
L_physics = mean(ReLU(−ŷ)²) + mean(ReLU(ŷ − 1)²)
```

- `ReLU(−ŷ)²`: penalises negative normalised predictions
  (wave front and velocity are physically non-negative).
- `ReLU(ŷ − 1)²`: penalises predictions outside the training range
  (encourages conservative extrapolation).

λ = 0.1 balances the data fit with physical plausibility.

### 4.4 Optimisation

| Hyperparameter | Value |
|----------------|-------|
| Optimiser      | Adam |
| Initial LR     | 1×10⁻³ |
| LR Schedule    | Cosine annealing (η_min = 1×10⁻⁵) |
| Batch size     | 32 |
| Max epochs     | 200 |
| Early stopping | patience = 20 epochs |

## 5. Evaluation Methodology

### 5.1 Metrics

All metrics are computed in physical units (not normalised) on the held-out
test set:

| Metric | Formula | Notes |
|--------|---------|-------|
| MAE    | mean\|ŷ − y\| | Intuitive unit-aware error |
| RMSE   | √mean(ŷ − y)² | Penalises large errors |
| R²     | 1 − SS_res/SS_tot | Variance explained |
| MaxAE  | max\|ŷ − y\| | Worst-case bound |

### 5.2 Visualisations

- **Parity plots**: predicted vs true for each output
- **Residual plots**: (ŷ − y) vs true, revealing systematic bias
- **Time series**: wave front and velocity curves over time for each case
- **Error distributions**: histograms of absolute errors
- **Loss curves**: train/val loss over epochs (log scale)

## 6. Known Limitations

### 6.1 Dataset Size

With only 3 cases × 26 time steps = 78 data points, the model is severely
data-constrained. Generalisation to unseen (H, W) pairs relies heavily on
the smooth physics. A production-grade surrogate would require ≥50 cases
(~1,300 data points minimum).

### 6.2 Output Scope

Only two scalar quantities are predicted. Full-field prediction (spatially
resolved α, U, p maps) would require:
- Convolutional architectures (U-Net) for structured grids
- Graph Neural Networks (GNN) for unstructured meshes
- Or Fourier Neural Operators (FNO) for physics-informed field learning

### 6.3 Physical Assumptions

- **2D laminar flow**: real dam-break events are 3D and turbulent at scale.
- **Flat bottom**: bathymetry, obstacles, or slopes are not accounted for.
- **No air compressibility**: entrained air is not modelled.
- **No surface tension**: σ = 0.07 N/m is included in OpenFOAM but the
  surrogate has no explicit surface tension input.

### 6.4 Temporal Generalisation

The time variable t enters as a raw input to the MLP. The model has no
explicit time-stepping structure, so it learns a static mapping
(H, W, t) → (x_front, v_max) rather than a dynamical system. This means
it cannot exploit temporal continuity and may produce non-monotonic wave
front predictions at extreme times.
